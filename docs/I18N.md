## 翻译文件结构

每种语言对应两个文件，放在 `src/assets/lang/`：

```
src/assets/lang/
├── zh-CN-ui.json        # UI 文本（导航、按钮、标签等，约 16KB）
├── zh-CN-product.json   # 产品文本（名称、参数、描述等，约 200KB）
├── en-ui.json
├── en-product.json
├── ...（其余 23 种语言）
└── languages.json       # 语言列表元数据
```

**分离设计的原因：**
- UI 文件小（16KB），首屏加载必须
- 产品文件大（200KB），按需加载，不影响首屏 TTI
- 两个文件可独立更新，互不干扰

## 翻译加载机制（TranslationManager）

### 加载流程

```
用户访问页面 / 切换语言
        ↓
loadTranslations(lang)
  ├── 命中 translationsCache → 直接返回（无 fetch）
  ├── 命中 pendingLoads → 等待同一 Promise（防并发重复请求）
  └── fetchTranslations(lang)
        ├── fetch ./assets/lang/{lang}-ui.json
        │     失败 → fallback fetch zh-CN-ui.json
        ├── fetch ./assets/lang/{lang}-product.json
        │     返回空 {} → fallback fetch zh-CN-product.json
        │     失败 → fallback fetch zh-CN-product.json
        ├── mergeTranslations(ui, product) → { ...ui, ...product }
        └── 写入 translationsCache，后续命中缓存直接返回
```

### 翻译键使用（前端 JS）

```javascript
// 所有翻译通过全局 tr() 函数调用（挂载在 window）
tr('nav_products', 'Products')
// → 当前语言有该键则返回翻译值，否则返回第二个参数作为 fallback

// 产品字段专用（支持 i18n 嵌套对象）
getProductI18nField(product, 'name', product.name)
```

### 翻译键使用（HTML 模板）

```html
<!-- 静态文本 -->
<span data-i18n="nav_products">Products</span>

<!-- placeholder 属性 -->
<input data-i18n-placeholder="search_placeholder" placeholder="Search...">

<!-- aria 属性 -->
<button data-i18n-aria-label="btn_close" aria-label="Close">×</button>
```

语言切换时，`TranslationManager` 通过 `MutationObserver` + `cachedElements` 批量更新所有带 `data-i18n` 属性的 DOM 元素。

### 语言初始化规则

```
1. 读取 localStorage.getItem('userLanguage')
2. 验证该代码在 lang-registry 的 languageNames 中存在
3. 有效 → 使用该语言
4. 无效或不存在 → 默认 zh-CN
```

## i18n 工具链命令

### 完整数据流

```
飞书多维表格（产品数据源）
  ↓ npm run sync:feishu
src/assets/product-data-table.js（产品原始数据）
  ↓ npm run i18n:extract
scripts/product-strings.json（提取出的 i18n 键）
  ↓ npm run product:sync:source
src/assets/lang/{lang}-product.json（中文源写入各语言文件）
  ↓ npm run translate:products:incremental
src/assets/lang/{lang}-product.json（Gemini API 翻译填充）
  ↓ npm run product:sync
src/assets/lang/{lang}-product.json（同步最终产品翻译）
  ↓ npm run product:collect
src/assets/lang/{lang}-product.json（按语言收集汇总）
  ↓ npm run split:lang
src/assets/lang/{lang}-ui.json + {lang}-product.json（拆分完成）
  ↓ webpack build
dist/assets/lang/（生产产物）
```

### 命令说明

```bash
# 从产品数据表提取 i18n 键（生成 scripts/product-strings.json）
npm run i18n:extract

# 合并所有语言翻译文件
npm run merge:i18n

# 按语言拆分为 {lang}-ui.json + {lang}-product.json
npm run split:lang

# 收集各语言产品翻译到统一文件
npm run product:collect

# 同步产品 i18n（仅同步中文源，不翻译）
npm run product:sync:source

# 同步产品 i18n（同步所有语言）
npm run product:sync

# 增量翻译（只翻译新增/变更的键）— 日常推荐
npm run translate:products:incremental

# 全量翻译（重新翻译全部键，耗时较长）
npm run translate:products
```

## 翻译引擎

### Gemini API 配置

翻译引擎封装在 `scripts/unified-translator.js`（60KB），通过 Gemini 3 API 实现批量多语言翻译。

配置方式（环境变量）：
```bash
GEMINI_API_KEY=your_api_key
GEMINI_BASE_URL=https://api.kuai.host   # 可选，默认官方地址
```

### 品牌词保护机制

`scripts/product-translation-handler.js` 在翻译前保护以下内容不被翻译：

- **品牌/产品型号**：ESL、ECOMAX、YuKoLi 等 20+ 品牌词
- **数字与单位**：`25kW`、`380V`、`50Hz`、`10-20kg/锅次` 等
- **Emoji**：直接保留原样
- **规格参数**：尺寸、重量等包含数字的字段

保护方式：翻译前用占位符替换敏感内容，翻译后还原。

### 增量翻译策略

`--incremental` 模式通过对比 `scripts/product-strings.json` 的哈希值，只翻译：
1. 新增的键
2. 中文原文发生变化的键

未变化的键直接复用现有翻译，大幅减少 API 调用量和翻译成本。

## 新增语言

### 完整步骤

**第一步：在 `src/lang-registry.js` 注册新语言**

```javascript
{
  code: 'vi',                    // 语言代码（BCP 47 格式）
  nativeName: 'Tiếng Việt',     // 前端弹窗显示的原生名称
  englishName: 'Vietnamese',     // Gemini 翻译 Prompt 中使用的英文名称
  hasTranslation: true,          // true = 参与翻译引擎
  uiGroup: 'asian',              // 'common' | 'european' | 'asian' | 'rtl'
  sortOrder: 21,                 // 弹窗排序（数字越小越靠前）
}
```

**第二步：创建空的 UI 翻译文件**

```bash
# 以 zh-CN-ui.json 为模板，创建新语言 UI 文件
cp src/assets/lang/zh-CN-ui.json src/assets/lang/vi-ui.json
# 将 vi-ui.json 中的中文值清空（保留键），或直接跑翻译
```

**第三步：同步并翻译产品数据**

```bash
npm run product:sync:source      # 同步中文源到新语言产品文件
npm run translate:products:incremental  # 调用 Gemini API 翻译
```

**第四步：验证**

```bash
npm run split:lang               # 拆分文件
npm start                        # 启动开发服务器，切换到新语言验证
```

## 常见问题排查

### 语言切换后页面文本没有变化

1. 打开浏览器控制台，检查是否有 `Failed to load translations` 报错
2. 确认 `dist/assets/lang/` 或 `src/assets/lang/` 下存在对应文件：
   ```bash
   ls src/assets/lang/ | grep "^en-"
   # 应输出：en-product.json  en-ui.json
   ```
3. 确认 `src/lang-registry.js` 中该语言的 `hasTranslation: true`

### 翻译文件存在但产品名称显示中文

该语言的 `-product.json` 为空 `{}`，属于预期 fallback 行为。  
执行以下命令生成翻译：
```bash
npm run translate:products:incremental
```

### 新增了 i18n 键但页面不显示

1. 确认 HTML 中使用了 `data-i18n="key_name"` 属性
2. 确认该键已添加到 `zh-CN-ui.json`（作为数据源）
3. 执行 `npm run merge:i18n && npm run split:lang` 同步到其他语言

### RTL 语言（阿拉伯语/希伯来语）布局问题

`ar` 和 `he` 属于 `rtl` 分组，语言切换时会自动为 `<html>` 设置 `dir="rtl"`，CSS 通过 `[dir="rtl"]` 选择器处理布局镜像。
